# Scoping and estimating

When the user shares a project brief and asks for help sizing or pricing:

## Step 1 — Understand the work

Ask in one question (skip what's clear):
- **What's the deliverable?** One sentence.
- **How experienced is the user with this kind of work?** Affects the estimate.
- **Hourly or project-based?** If they have a number in mind.

## Step 2 — Break down the work

Split into 4-7 phases. For each:
- **Name** — verb-led ("Discovery", "Design", "Build", "Review")
- **Realistic estimate** — hours or days
- **Buffer** — noted explicitly, not hidden

## Step 3 — Output

```
Project: [short name]

Phases:
1. [Phase] — [X hours]
2. [Phase] — [Y hours]
3. [Phase] — [Z hours]
...

Subtotal: [N hours]
Buffer (recommended +25-40%): [M hours]
Total estimate: [N+M hours]

Pricing options:
- Hourly @ $[rate]: $[total]
- Project-based: $[total + 10-20% premium for fixed-price risk]

Risks to flag:
- [risk 1, e.g., "first-time client — relationship risk"]
- [risk 2, e.g., "scope described vaguely — clarify in the proposal"]
```

## Rules

- **Always include a buffer.** Most freelancers under-estimate by 30-50%. Pad explicitly so the user can decide whether to absorb it.
- **Surface 2-3 risks.** What could blow up the estimate?
- **Don't undercut the user.** If their proposed rate seems low, gently flag it — but ultimately respect their pricing call.
- **For project-based pricing, premium fixed-price work +10-20%** over hourly equivalent to compensate for scope risk.

End with: "Want me to write this up as a proposal, or run different rate scenarios?"
