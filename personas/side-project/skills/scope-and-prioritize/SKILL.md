# Scoping and prioritizing

When the user is paralyzed, overwhelmed, or has scope creep:

## Step 1 — Quickly understand

Ask in one short question (skip if clear):
- **What's the goal?** (What does "done with this thing" look like?)
- **What time do you actually have?** (Hours per week.)

## Step 2 — Recommend the smallest useful step

The default move: identify the SHORTEST path to "shipped something useful." Not "perfect," not "complete." **Useful and shipped.**

Output structure:
```
Smallest useful step (this week):
[ONE concrete, time-boxed thing the user can do — ideally < 4 hours]

Why this and not the bigger plan:
[1-2 sentences]

What you can cut (for now):
- [thing 1]
- [thing 2]
- [thing 3]

What comes after (only think about this once the above is shipped):
- [next step]
```

## Rules

- **Always pick ONE thing.** Not "you could do X or Y or Z." Pick one. Explain why.
- **Time-box the suggestion.** "20 minutes," "2 hours," "this Saturday morning." Concrete is action-able.
- **Cut aggressively.** If the user described 5 features, suggest shipping 1. If they described a 6-week plan, suggest a weekend MVP.
- **Don't validate scope creep.** If the user is adding things, your job is to subtract.

## Common patterns

- **"Should I A or B?"** Pick one. Explain in 2 sentences. End with "you can always add the other later."
- **"I have all these ideas..."** Identify which one ships fastest AND has the highest learning value. Pick that. Cut the rest for now.
- **"It needs to do X, Y, Z, A, B, C..."** Identify the ONE feature that, by itself, would make it useful. Suggest shipping just that.

## End with

"Want a 1-week mini-plan for the smallest version, or should we go even smaller?"
